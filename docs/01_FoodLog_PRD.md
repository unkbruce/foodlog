# FoodLog PRD

## 1. 프로젝트 개요

| 항목 | 내용 |
| --- | --- |
| 프로젝트명 | FoodLog |
| 서비스 유형 | 음식 기록 서비스 |
| 프로젝트 단계 | 프로젝트 초기 기획 및 화면 설계 단계 |
| 주요 기술 | Node.js, Express |
| 데이터 저장 방식 | DB 없이 서버 메모리의 임시 배열 사용 |

FoodLog는 사용자가 먹은 음식을 간단하게 기록하고, 기록한 음식 히스토리를 확인하며, 식단에 대한 기본적인 피드백을 받을 수 있는 음식 기록 서비스이다.  
현재 단계에서는 복잡한 회원 기능이나 데이터베이스 연동 없이, 음식 기록의 추가, 조회, 수정, 삭제 기능과 간단한 식단 피드백 흐름을 중심으로 MVP를 구성한다.

현재 음식 데이터 항목은 다음과 같다.

| 데이터 항목 | 개발 필드명 | 설명 |
| --- | --- | --- |
| 고유 번호 | id | 음식 기록 고유 번호 |
| 음식 이름 | name | 사용자가 기록한 음식 이름 |
| 카테고리 | category | 음식 종류 또는 분류 |
| 칼로리 | calories | 음식의 칼로리 |
| 메모 | memo | 음식에 대한 간단한 메모 |
| 기록 일시 | createdAt | 음식 기록이 저장된 일시 |

## 2. 서비스 기획 배경

일상에서 먹은 음식을 꾸준히 기억하고 관리하는 것은 쉽지 않다. 특히 식단 관리를 처음 시작하는 사용자는 어떤 음식을 먹었는지, 하루 동안 어느 정도의 칼로리를 섭취했는지, 식단이 한쪽으로 치우치지는 않았는지 확인하기 어렵다.

FoodLog는 이러한 문제를 해결하기 위해 사용자가 먹은 음식을 빠르게 기록하고, 기록된 데이터를 기반으로 간단한 피드백을 제공하는 것을 목표로 한다. 프로젝트 초기 단계에서는 복잡한 분석 기능보다 음식 기록 흐름, 화면 구성, 사용자 시나리오를 명확히 잡는 데 초점을 둔다.

## 3. 문제 정의

| 문제 | 설명 |
| --- | --- |
| 음식 기록의 번거로움 | 사용자가 매번 복잡한 입력을 해야 하면 기록을 지속하기 어렵다. |
| 식단 히스토리 확인 어려움 | 별도로 기록하지 않으면 과거에 무엇을 먹었는지 확인하기 어렵다. |
| 간단한 식단 판단 기준 부족 | 사용자는 자신의 식단이 적절한지 판단하기 어려울 수 있다. |
| 프로젝트 범위 관리 필요 | 초기 버전에서는 핵심 기능 중심으로 범위를 정리해야 한다. |

FoodLog는 위 문제를 해결하기 위해 음식 기록 추가, 목록 조회, 상세 조회, 수정, 삭제 기능을 제공하고, 기록된 데이터를 바탕으로 간단한 규칙 기반 피드백을 제공한다.

## 4. 타깃 사용자

| 사용자 유형 | 설명 |
| --- | --- |
| 식단 기록을 처음 시작하는 사용자 | 복잡한 기능보다 간단한 음식 기록 기능이 필요한 사용자 |
| 하루 식사 내용을 확인하고 싶은 사용자 | 자신이 먹은 음식 목록과 칼로리를 간단히 확인하려는 사용자 |
| 개인 식단 기록을 간단히 관리하고 싶은 사용자 | 복잡한 관리 기능보다 기본 기록과 확인 기능을 원하는 사용자 |

## 5. 서비스 목표

1. 사용자가 먹은 음식을 쉽고 빠르게 기록할 수 있도록 한다.
2. 기록된 음식 목록을 통해 식단 히스토리를 확인할 수 있도록 한다.
3. 음식 기록을 상세 조회하고 필요할 때 수정 또는 삭제할 수 있도록 한다.
4. 기록된 음식 데이터를 기준으로 간단한 식단 피드백을 제공한다.
5. 핵심 기능 중심으로 이해하기 쉬운 구조와 화면 흐름을 구성한다.

## 6. 핵심 기능

### 6.1 음식 기록 추가

사용자가 음식 이름, 카테고리, 칼로리, 메모를 입력하여 새로운 음식 기록을 추가하는 기능이다.

| 항목 | 내용 |
| --- | --- |
| 처리 내용 | 입력한 음식 정보를 새 기록으로 저장 |
| 입력 데이터 | 음식 이름, 카테고리, 칼로리, 메모 |
| 생성 데이터 | 고유 번호, 음식 이름, 카테고리, 칼로리, 메모, 기록 일시 |
| 주요 목적 | 사용자의 음식 섭취 기록 저장 |

### 6.2 음식 기록 목록 조회

사용자가 지금까지 등록한 음식 기록 전체를 목록으로 확인하는 기능이다.

| 항목 | 내용 |
| --- | --- |
| 처리 내용 | 저장된 음식 기록 목록을 불러와 표시 |
| 표시 데이터 | 음식 이름, 카테고리, 칼로리, 기록 일시 |
| 주요 목적 | 전체 음식 히스토리 확인 |

### 6.3 음식 기록 상세 조회

사용자가 특정 음식 기록을 선택하여 상세 정보를 확인하는 기능이다.

| 항목 | 내용 |
| --- | --- |
| 처리 내용 | 선택한 음식 기록의 상세 정보를 표시 |
| 표시 데이터 | 고유 번호, 음식 이름, 카테고리, 칼로리, 메모, 기록 일시 |
| 주요 목적 | 개별 음식 기록의 상세 내용 확인 |

### 6.4 음식 기록 수정

사용자가 기존 음식 기록의 이름, 카테고리, 칼로리, 메모를 수정하는 기능이다.

| 항목 | 내용 |
| --- | --- |
| 처리 내용 | 선택한 음식 기록의 수정 내용을 저장 |
| 수정 가능 데이터 | 음식 이름, 카테고리, 칼로리, 메모 |
| 주요 목적 | 잘못 입력한 음식 기록 수정 |

### 6.5 음식 기록 삭제

사용자가 더 이상 필요하지 않은 음식 기록을 삭제하는 기능이다.

| 항목 | 내용 |
| --- | --- |
| 처리 내용 | 선택한 음식 기록을 삭제 |
| 주요 목적 | 불필요하거나 잘못 등록된 기록 제거 |

### 6.6 식단 피드백

기록된 음식 데이터를 기준으로 사용자에게 간단한 피드백을 제공하는 기능이다. 현재 MVP에서는 AI 분석이 아닌 규칙 기반 피드백으로 구성한다.

예시 피드백 규칙은 다음과 같다.

| 조건 | 피드백 예시 |
| --- | --- |
| 총 칼로리가 높은 경우 | 오늘 섭취 칼로리가 높은 편이므로 가벼운 식사를 고려한다. |
| 특정 카테고리에 기록이 집중된 경우 | 식단이 한 종류에 치우칠 수 있으므로 다양한 음식을 섭취한다. |
| 기록된 음식이 적은 경우 | 아직 기록이 적으므로 식사 후 꾸준히 기록한다. |

## 7. 사용자 시나리오

| 단계 | 사용자 행동 | 시스템 동작 |
| --- | --- | --- |
| 1 | 사용자가 음식 기록 목록 화면에 접속한다. | 저장된 음식 기록 목록을 보여준다. |
| 2 | 사용자가 음식 추가 버튼을 선택한다. | 음식 기록 추가 화면으로 이동한다. |
| 3 | 사용자가 음식 이름, 카테고리, 칼로리, 메모를 입력한다. | 입력값을 받아 새 음식 기록을 생성한다. |
| 4 | 사용자가 등록을 완료한다. | 목록 화면에 새 음식 기록을 반영한다. |
| 5 | 사용자가 특정 음식 기록을 선택한다. | 상세 화면에서 음식 정보를 보여준다. |
| 6 | 사용자가 내용을 수정하거나 삭제한다. | 수정된 내용을 저장하거나 해당 기록을 삭제한다. |
| 7 | 사용자가 식단 피드백 화면을 확인한다. | 기록된 음식 데이터를 기준으로 간단한 피드백을 보여준다. |

## 8. 주요 화면 구성

### 8.1 음식 기록 목록 화면

음식 기록 전체를 확인하는 메인 화면이다.

| 구성 요소 | 설명 |
| --- | --- |
| 음식 목록 | 등록된 음식 기록을 리스트 형태로 표시 |
| 음식 정보 요약 | 음식 이름, 카테고리, 칼로리, 기록 일시 표시 |
| 추가 버튼 | 음식 기록 추가 화면으로 이동 |
| 상세 이동 | 선택한 음식의 상세/수정 화면으로 이동 |
| 피드백 이동 | 식단 피드백 화면으로 이동 |

### 8.2 음식 기록 추가 화면

새로운 음식 기록을 입력하는 화면이다.

| 구성 요소 | 설명 |
| --- | --- |
| 음식 이름 입력 | 사용자가 먹은 음식 이름 입력 |
| 카테고리 입력 | 음식 종류 또는 분류 입력 |
| 칼로리 입력 | 음식의 칼로리 입력 |
| 메모 입력 | 식사 상황, 느낌, 참고 내용 입력 |
| 등록 버튼 | 입력한 내용을 저장 |
| 취소 버튼 | 목록 화면으로 돌아가기 |

### 8.3 음식 기록 상세/수정 화면

선택한 음식 기록의 상세 정보를 확인하고 수정할 수 있는 화면이다.

| 구성 요소 | 설명 |
| --- | --- |
| 상세 정보 표시 | 음식 이름, 카테고리, 칼로리, 메모, 기록 일시 표시 |
| 수정 입력 영역 | 기존 음식 정보를 수정할 수 있는 입력 폼 |
| 저장 버튼 | 수정된 내용을 반영 |
| 삭제 버튼 | 해당 음식 기록 삭제 |
| 뒤로가기 버튼 | 목록 화면으로 이동 |

### 8.4 식단 피드백 화면

기록된 음식 데이터를 바탕으로 간단한 식단 피드백을 제공하는 화면이다.

| 구성 요소 | 설명 |
| --- | --- |
| 총 기록 수 | 현재 등록된 음식 기록 개수 표시 |
| 총 칼로리 | 기록된 음식의 칼로리 합계 표시 |
| 카테고리 요약 | 음식 카테고리별 기록 현황 표시 |
| 피드백 메시지 | 규칙 기반 식단 피드백 표시 |

## 9. MVP 범위

FoodLog의 MVP는 핵심 음식 기록 기능과 기본 피드백 기능을 확인할 수 있는 범위로 제한한다.

| 구분 | 포함 여부 | 설명 |
| --- | --- | --- |
| 음식 기록 추가 | 포함 | 입력한 음식 정보를 새 기록으로 저장 |
| 음식 기록 목록 조회 | 포함 | 저장된 음식 기록 목록 확인 |
| 음식 기록 상세 조회 | 포함 | 선택한 음식 기록의 상세 정보 확인 |
| 음식 기록 수정 | 포함 | 선택한 음식 기록의 내용 수정 |
| 음식 기록 삭제 | 포함 | 불필요한 음식 기록 삭제 |
| 식단 피드백 | 포함 | 규칙 기반 간단 피드백 |
| 회원가입/로그인 | 제외 | MVP 범위에서는 구현하지 않음 |
| 데이터베이스 연동 | 제외 | 서버 메모리의 임시 배열 사용 |
| AI API 연동 | 제외 | 향후 확장 방향으로 정의 |

## 10. 향후 확장 방향

현재 FoodLog는 규칙 기반 피드백을 사용한다. 예를 들어 총 칼로리, 음식 카테고리 분포, 기록 개수 등을 기준으로 미리 정한 문구를 보여주는 방식이다.

향후에는 AI API를 연동하여 개인화된 식단 피드백 기능으로 확장할 수 있다. 사용자의 음식 기록 데이터를 AI API에 전달하고, 식단 균형, 칼로리 섭취 경향, 개선 제안 등을 자연어로 제공하는 방식으로 발전시킬 수 있다.

확장 가능 기능은 다음과 같다.

| 확장 기능 | 설명 |
| --- | --- |
| 데이터베이스 연동 | 음식 기록을 서버 재시작 후에도 유지 |
| 회원 기능 | 사용자별 음식 기록 관리 |
| 날짜별 필터 | 특정 날짜의 음식 기록만 조회 |
| 통계 기능 | 일별, 주별, 카테고리별 칼로리 통계 제공 |
| AI 식단 피드백 | AI API를 활용한 개인화 피드백 제공 |
| 이미지 업로드 | 음식 사진을 함께 기록 |

## 11. 기대 효과

FoodLog를 통해 사용자는 자신이 먹은 음식을 간단히 기록하고, 기록된 히스토리를 통해 식단을 되돌아볼 수 있다. 또한 기본적인 피드백을 통해 식단 관리에 대한 관심을 높일 수 있다.

프로젝트 관점에서는 Node.js와 Express를 활용해 음식 기록 기능, 기본 기록 관리 흐름, 화면 기획, 사용자 시나리오를 함께 구성할 수 있다. 기능 범위를 핵심 중심으로 유지하기 때문에 전체 서비스 흐름을 파악하기 쉽고, 이후 데이터베이스나 AI API 연동으로 확장하기에도 적합하다.
